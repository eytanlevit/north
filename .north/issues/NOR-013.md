**Context**: Enable coding agents to autonomously claim issues, track their work, and mark completion, preventing conflicts when multiple agents work in parallel.

**Requirements**:
- Agent can "acquire" an issue (update status to in-progress, set assignee)
- Track which agent is working on which issue (show in UI)
- Store agent metadata: assignee ID, worktree path, tmux session, start timestamp
- Agent must provide completion summary when marking issue done
- Block claiming if issue already assigned (show error, don't allow override)
- Display agent assignment in issue list as label: `[agent-work-1] NOR-005 Rename project...`
- In issue detail view, provide keyboard shortcut (press 'a') to attach to agent's tmux session

**Acceptance Criteria**:
- Agent can call `update_issue` to claim an issue (sets assignee + in-progress)
- UI shows agent name as label in issue list
- Attempting to claim an already-assigned issue shows error and blocks
- Agent must provide summary when marking done (required field)
- User can press 'a' in issue detail to attach to tmux session

**Technical Details**:
- Files: `src/issues.ts` (add assignee fields), `src/components/kanban-pane.ts`, `issue-detail.ts`
- Add to issue frontmatter:
  ```yaml
  assignee: "agent-work-1"
  started_at: "2026-03-01T18:30:00Z"
  worktree: ".worktrees/NOR-005"
  tmux_session: "NOR-005"
  ```
- Create agent workflow tools:
  - `acquire_issue(id)` - claim issue, error if already assigned
  - `complete_issue(id, summary)` - mark done (summary required), add comment with summary
- UI changes:
  - Issue list: show `[agent-name]` label before issue title
  - Issue detail: show assignee info, add 'a' keybinding to run `tmux attach -t {session}`
- Validation: check assignee field before allowing claim, return error if set

**Out of Scope**:
- Automatic agent scheduling (which agent gets which issue)
- Agent performance metrics or time tracking
- Override/steal functionality (strict blocking for now)
