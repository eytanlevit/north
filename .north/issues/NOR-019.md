Ensure pmtui works correctly with git worktrees - handle .pm folder presence/absence and decide on git tracking strategy.

## Problem
When creating git worktrees for parallel agent work (see NOR-012), the .pm folder behavior is unclear:

**Scenario 1: .pm is gitignored (current?)**
- New worktree won't have .pm folder
- Agent can't see or update issues
- Each worktree needs its own .pm? (issues diverge)

**Scenario 2: .pm is tracked in git**
- New worktree gets .pm folder
- But issues are tracked in git
- Multiple agents modifying issues = merge conflicts
- Concurrent issue updates problematic

**Scenario 3: Duplicate .pm folders**
- Each worktree has independent .pm
- Issues don't sync between worktrees
- Main tree and worktrees out of sync

## Options to Evaluate

### Option A: Shared .pm (symlink)
```bash
# When creating worktree for NOR-012
git worktree add .worktrees/NOR-012
cd .worktrees/NOR-012
rm -rf .pm
ln -s ../../.pm .pm
```
**Pros:**
- All worktrees share same .pm folder
- No conflicts, always in sync
- Simple to understand

**Cons:**
- Symlinks in git repos can be problematic
- Not cross-platform (Windows?)
- Worktree deletion doesn't clean up symlink

### Option B: Track in git + conflict handling
```bash
# Track .pm in git
git add .pm/
echo ".pm/chats/" >> .gitignore
```
**Pros:**
- Issues tracked in git (versioned, backed up)
- Team collaboration works naturally
- Worktrees automatically get .pm
- Git handles most of the sync

**Cons:**
- Merge conflicts possible
- Need to handle concurrent updates
- Session logs tracked or ignored?

### Option C: Centralized issue store outside git
```bash
# .pm lives outside git entirely
~/.pmtui/projects/pmtui/.pm/
```
**Pros:**
- No conflicts ever
- All worktrees reference same location
- Not versioned, but always consistent

**Cons:**
- Issues not versioned or backed up
- Doesn't work with team collaboration
- How to identify which project?

### Option D: Hybrid approach
- Track `.pm/config.yaml` and `.pm/project.md` in git
- Keep `.pm/issues/` outside git (shared or centralized)
- Session logs outside git

**Pros:**
- Config versioned, issues conflict-free
- Worktrees share issue state

**Cons:**
- Complex to explain and implement
- Issues not backed up in git
