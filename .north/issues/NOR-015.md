Currently project name and description exist in both `config.yaml` and `project.md`, creating potential inconsistencies. Consolidate to a single source of truth.

## Current State
- `config.yaml` has `name` and `description` fields
- `project.md` has project name in heading and full description in body
- Easy for these to get out of sync
- Duplication of information

## Proposed Solution
**Option 1: Config.yaml as source of truth**
- Keep name and description in config.yaml
- project.md becomes free-form notes only
- UI reads from config.yaml
- Simpler, structured data

**Option 2: Project.md as source of truth**
- Remove name/description from config.yaml
- Parse project.md for name (# heading) and description
- More markdown-native approach
- Allows richer formatting in description

**Option 3: Hybrid**
- Keep short description in config.yaml (one-liner)
- Full documentation in project.md
- Config description is a summary, project.md is detailed

## Recommendation
**Option 1** seems cleanest - config.yaml owns the canonical name/description, project.md is for extended notes and documentation.

## Implementation
- Decide on single source of truth
- Update project detail view to read from correct location
- Update show_project and update_project tools
- Migration: if both exist, prefer one and warn/migrate the other
- Update documentation

## Files to Update
- `src/config.ts`
- `src/tools/show-project.ts`
- `src/tools/update-project.ts`
- `src/components/project-detail.ts`
