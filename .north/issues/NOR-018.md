Add integration to sync issues bidirectionally with external issue tracking systems (GitHub Issues, Linear), with the option to use them as the source of truth instead of local filesystem.

## Two Modes

### Mode 1: Local Source of Truth (Sync)
- Issues stored locally in `.pm/issues/`
- Sync to GitHub/Linear for visibility
- Local changes push to external platforms
- External changes pull to local

### Mode 2: External Source of Truth
- GitHub/Linear is the canonical source
- Local `.pm/issues/` is a cache/mirror
- All reads/writes go through API
- No local filesystem as source of truth
- Configured during `pmtui init`

## Init Configuration
During project initialization, prompt:
```
Where should issues be stored?

1. Local filesystem (.pm/issues/)
   - Issues stored as markdown files
   - Git-tracked, versioned
   - Works offline
   - Optional sync to GitHub/Linear

2. GitHub Issues
   - Repository: [enter repo: user/repo]
   - Token: [enter or use $GITHUB_TOKEN]
   - Issues fetched from GitHub API
   - All changes made via API

3. Linear
   - Team: [enter team ID]
   - Token: [enter or use $LINEAR_TOKEN]  
   - Issues fetched from Linear API
   - All changes made via API
```

## Configuration
```yaml
# .pm/config.yaml
storage:
  type: "github"  # or "local" or "linear"
  
integrations:
  github:
    enabled: true
    source_of_truth: true  # this is the canonical source
    repo: "user/repo"
    token: "${GITHUB_TOKEN}"
    cache_locally: true  # optional: cache for offline access
  
  linear:
    enabled: false
    team: "TEAM"
    token: "${LINEAR_TOKEN}"
```

## Benefits of External Source of Truth
- ✅ Team immediately sees changes (no sync delay)
- ✅ No local/remote conflicts
- ✅ Leverage platform features (GitHub Projects, Linear cycles)
- ✅ Existing issues already there (no import needed)
- ✅ Single source of truth for team
- ✅ Platform handles permissions, access control

## Challenges of External Source of Truth
- ❌ Requires internet connection
- ❌ API rate limiting
- ❌ Slower than local filesystem
- ❌ Dependent on external service uptime
- ❌ May need local caching for performance

Create an `IssueStore` interface:
```typescript
interface IssueStore {
  listIssues(): Promise<Issue[]>;

  deleteIssue(id: string): Promise<void>;
}

class LocalFileStore implements IssueStore { ... }
class GitHubStore implements IssueStore { ... }
class LinearStore implements IssueStore { ... }
```

### Hybrid Mode (Optional)

- Local cache for offline access

## Mode 1: Local + Sync (Existing Plan)

### Requirements
- Sync issues between pmtui and external platforms
- Bidirectional sync: changes in either direction are reflected
- Support for GitHub Issues
- Support for Linear
- Map pmtui fields to external platform fields
- Handle conflicts and sync state

### Use Cases
- Keep pmtui as local source of truth while syncing to GitHub for team visibility
- Import existing GitHub/Linear issues into pmtui
- Update issue status in pmtui, sync to external platform
- Collaborate with team members who use GitHub/Linear

### Field Mapping Considerations
#### GitHub Issues
- title ↔ title
- body ↔ body
- status → state (open/closed) - need custom mapping for in-progress
- priority → labels? (high/medium/low as labels)
- comments ↔ comments
- labels ↔ labels
- blocked_by → ? (maybe reference in body or custom field)

#### Linear
- title ↔ title
- body ↔ description
- status ↔ state (better status support)
- priority ↔ priority (native support)
- comments ↔ comments
- labels ↔ labels
- parent → parent issue
- blocked_by → blocked by relations

### Technical Considerations
- Authentication (GitHub token, Linear API key)
- Rate limiting
- Sync strategy: polling vs webhooks
- Conflict resolution (last write wins? manual merge?)
- Offline mode - queue changes when offline
- Issue ID mapping (external ID stored in frontmatter?)
- Initial import vs continuous sync

### Issue Metadata (for sync mode)
Add to frontmatter:
```yaml
external_ids:
  github: 123
  linear: "ISS-abc123"
last_synced: "2026-03-01T18:30:00Z"
```

### Commands/Tools
- `/sync` - manual sync trigger
- `pmtui sync` - CLI command to sync
- Auto-sync on issue create/update (configurable)

### Challenges
- Status mapping (pmtui has 3 states, platforms may have more/less)
- Handling platform-specific features (GitHub projects, Linear cycles)
- Performance with large issue counts
- Preventing sync loops
- Partial failures (some issues sync, others fail)
